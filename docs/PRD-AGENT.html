1. Visi dan Tujuan Produk

1.1 Pernyataan Visi

CAKRA Intelligence Platform adalah platform operasi terpadu berbasis web yang menyatukan pelacakan geolokasi real-time, komunikasi tim, analisis kecerdasan buatan, dan integrasi video surveillance dalam satu antarmuka — memberikan Common Operating Picture (COP) kepada seluruh pemangku kepentingan operasi.

1.2 Masalah yang Diselesaikan & Tujuan Bisnis

  • Mengatasi silo data operasional dengan memusatkan koordinat GPS, aliran RTSP CCTV/Drone, data intelijen, dan chat terenkripsi ke dalam satu state management (Zustand/Redux) di frontend.
  • Mempercepat waktu respons komando dengan latensi websocket < 2 detik.
  • Mengotomatisasi analisis probabilitas dan heat map menggunakan integrasi REST API dengan model Pihak Kedua.

4. Alur Integrasi (System Flow)

  1. Data Ingestion: GPS dari mobile app (WSS) → Load Balancer → NestJS Microservice → Redis Pub/Sub → Disalurkan ke semua client tersambung di room terkait.
  2. AI Integrations: Setiap request analisis prediksi diteruskan (Proxy) oleh backend NestJS ke server AI Pihak Kedua untuk menghindari isu CORS dan melindungi API Key.
  3. Video Pipeline: Hardware Drone (RTMP) → K8s Ingress (Port 1935) → Node-Media-Server Pod → Chunking ke .ts/.m3u8 (HLS) → Disajikan via Nginx (Port 443) ke video player frontend (Video.js).

5. Kebutuhan Non-Fungsional (Arsitektur & Keamanan)

AspekDetail Implementasi & Standar
Performa APIResponse time REST API < 200ms (P95). WebSocket latency < 50ms internal cluster.
SkalabilitasArsitektur stateless. Backend NestJS dijalankan dalam Kubernetes Deployment dengan HPA (Horizontal Pod Autoscaler). Session state disimpan di Redis.
Media StreamingNode-Media-Server dijalankan di node pool terpisah dengan alokasi bandwidth dan CPU tinggi (compute-optimized).
Keamanan AuthAuthentication via JWT. Refresh Token di-rotate secara otomatis. Password di-hash dengan algoritma Argon2 (argon2id).
Keamanan DataTransport layer TLS 1.3 (Let's Encrypt). Komunikasi antar service dalam K8s diisolasi menggunakan Network Policies.
ObservabilityMetrics dikumpulkan oleh Prometheus. Logs diagregasi oleh Loki. Dashboard monitoring menggunakan Grafana (melihat CPU/RAM/Network/Active Sockets).
Offline ResilienceFrontend PWA menggunakan Service Workers untuk caching asset statis. Request queue menggunakan algoritma exponential backoff saat sinkronisasi ulang.

6. Metrik Keberhasilan (Engineering QA)

  • Coverage Test: Unit Test coverage backend minimal 70% (menggunakan Jest).
  • Load Testing: Sistem harus sanggup menangani 200 koneksi WebSocket simultan dengan emit data 5s/koneksi tanpa memory leak (via k6/Artillery).
  • GPS Deviasi: Akurasi parser geometri dari PostGIS tidak mengalami floating point truncation (> 6 decimal places).

7. Dependensi dan Risiko Teknis

7.1 Dependensi & Blocking Issues

  • Dokumentasi API Pihak Kedua (Crowd, Telco, Prediction) harus dikirim sebelum Sprint 3. (Blocking fitur P1).
  • Node-Media-Server membutuhkan alokasi storage I/O tinggi (NVMe) untuk transmuxing HLS tanpa buffering.

7.2 Mitigasi Risiko Teknis

  • Risiko: AI Endpoint Pihak Kedua mati (Downtime).
    Mitigasi: Backend NestJS harus menerapkan Circuit Breaker (menggunakan library `opossum`). Jika error beruntun, berikan respons fallback statis ke frontend agar peta tidak hang.
  • Risiko: Ledakan memori karena antrean Redis Pub/Sub.
    Mitigasi: Terapkan max-memory policy `allkeys-lru` di Redis dan lakukan rate limiting 1 emit per detik per user di sisi frontend.
